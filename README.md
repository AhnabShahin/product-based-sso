# WP Cross-Domain SSO Plugin

## Goal
Build a secure WordPress plugin that lets users switch between WordPress sites (products) seamlessly. When a user switches to a product, they are automatically logged in on that site using an `auth_key` that is valid for 30 seconds and bound to their device and browser.

This plugin treats each WordPress site as a **product** with its own `web_key`. Users are authenticated using short-lived, device-bound `auth_key` tokens that expire and auto-delete after 30 seconds.

## Core Concepts

- **Product**: A WordPress site with a name, logo, page URL, and a unique `web_key`.
- **Web Key**: A secret generated by the admin on each site using a PIN or code. Used to sign `auth_key` tokens.
- **Auth Key**: A short-lived token generated when a user switches to a product. Contains user email, IP, device details, browser details, and an expiry time of 30 seconds.
- **Device Binding**: `auth_key` includes device fingerprint and browser details. Token is rejected if device/browser changes.
- **Auto-cleanup**: `auth_key` tokens are stored in the database and auto-deleted 30 seconds after creation.

## Architecture Flow

1. **Source Site (Current Product)**: User clicks "Switch to [Product]".
2. **Generate Auth Key**: 
   - Use product's `web_key` to sign auth data
   - Include: email, IP, device details, browser details
   - Set expiry to: now + 30 seconds
3. **Redirect**: Send user to target product's page URL with `auth_key` in the query string.
4. **Target Site (Destination Product)**:
   - Check if `auth_key` exists in database
   - If YES: Redirect to page URL immediately (no login/logout needed)
   - If NO: Validate all details, logout current user, find user by email, log them in, store `auth_key` in database
5. **Auto-cleanup**: After 30 seconds, `auth_key` is auto-deleted from database.

## Step-by-Step Build Plan

### 1. Create the plugin skeleton

Create a new plugin folder named `product-based-sso` with a standard WordPress structure:

- `product-based-sso.php` for the plugin bootstrap
- `src/` for PHP classes
- `src/Admin/` for admin UI and settings pages
- `src/Services/` for auth key generation and validation
- `src/Controllers/` for request handling and redirects
- `src/Repositories/` for products and auth keys storage helpers
- `ui-src-assets/` for source CSS/JS assets
- `ui-src-build/` for built/minified assets and bundles
- `languages/` for translation files if needed

### 2. Define the data model

The current website is a product. Users (admins) can add other sites as products.

**Product attributes:**
- Product ID
- Name (e.g., "Main Site", "Blog", "Shop")
- Logo URL
- Page URL (redirect destination after login)
- Web Key (unique secret generated from PIN/code)
- Active/Inactive status
- Created/Updated timestamps

**Web Key:**
- Auto-generated by admin using a PIN or secret code
- Unique per product
- Used to sign auth keys
- Can be regenerated

### 3. Generate web keys on each site

Each WordPress site running this plugin should generate its own `web_key`:

- Admin navigates to SSO settings
- Clicks "Generate Web Key"
- Enters a PIN or code (or let the plugin auto-generate one)
- The plugin creates a secure `web_key` and stores it in WordPress options
- This `web_key` is used to sign all outgoing `auth_key` tokens

### 4. Build the admin products page

Add a settings page where admins can:

- View the current site as a product
- View the current site's `web_key`
- Add a new product (remote site)
  - Product name
  - Product logo URL
  - Product page URL (where user will be redirected after login)
  - Generate or paste a `web_key` from the remote site
- Enable or disable each product
- Delete a product
- View recent authentication attempts and logs

### 5. Build the frontend switch button

When a logged-in user is on the current site:

- Show a list of available products
- For each product, show a "Switch to [Product Name]" button
- Include the product logo
- On click, trigger a switch action that generates an `auth_key` and redirects

### 6. Implement auth key generation

The source site (current product) should:

1. Verify the user is logged in
2. Gather auth data:
   - User email
   - User IP address
   - Device fingerprint (browser + OS + screen resolution + etc.)
   - Browser user-agent
   - Current timestamp
3. Build a payload:
   ```json
   {
     "email": "user@example.com",
     "ip": "192.168.1.1",
     "device_fingerprint": "abc123def456",
     "user_agent": "Mozilla/5.0...",
     "product_id": 2,
     "from_product": "main-site",
     "to_product": "target-site",
     "iat": 1710000000,
     "exp": 1710000030
   }
   ```
4. Sign the payload:
   - JSON-encode the payload in a stable way
   - Generate HMAC SHA-256 signature using the **target product's web_key**
5. Combine into auth_key:
   ```json
   {
     "payload": {...},
     "signature": "xyz789"
   }
   ```
6. Base64-encode the final auth_key
7. Redirect to: `https://target-product-url?auth_key=BASE64_ENCODED_KEY`

### 7. Implement auth key validation on target site

The target site receives the `auth_key` at `/sso` endpoint:

1. **Check if auth_key exists in database:**
   - Query `wp_sso_auth_keys` for this exact `auth_key`
   - If found: Redirect to the target product's page URL immediately (do not re-validate)
   - If not found: Continue to step 2

2. **Decode and validate the token:**
   - Confirm request is HTTPS only
   - Base64-decode the auth_key
   - Verify payload and signature exist
   - Recompute HMAC-SHA256 using the **current site's web_key**
   - Compare signatures using constant-time comparison

3. **Validate token contents:**
   - Confirm token is not expired: `exp >= now`
   - Confirm target product matches current site
   - Confirm source product is in trusted products list
   - Verify device fingerprint matches current visitor
   - Verify IP address matches (or allow tolerance based on settings)
   - Verify user-agent matches (or allow tolerance based on settings)

4. **Find and log in user:**
   - Query WordPress users by email
   - If not found: Log failure and reject
   - If found: 
     - Logout any currently logged-in user on this site
     - Call `wp_set_current_user($user->ID)`
     - Call `wp_set_auth_cookie($user->ID, true)`
     - Trigger `do_action('wp_login', $user->user_login, $user)`

5. **Store auth_key in database:**
   - Insert into `wp_sso_auth_keys` table with:
     - `auth_key` (hashed for security)
     - `user_id`
     - `email`
     - `ip`
     - `device_fingerprint`
     - `user_agent`
     - `source_product`
     - `target_product`
     - `expires_at` (now + 30 seconds)
     - `created_at`
   - This entry will be checked on subsequent requests

6. **Redirect to page URL:**
   - Redirect to the target product's page URL

### 8. Auto-cleanup expired auth keys

Implement a cleanup mechanism:

- **Option 1 (Simple):** Use WordPress transients with 30-second TTL instead of database storage
- **Option 2 (Explicit):** Use a database table and run a cleanup job:
  - On every SSO request, delete any auth_keys where `expires_at < NOW()`
  - Or use a scheduled wp-cron job to clean up every minute
  - Or use a simple query on first-save: `DELETE FROM wp_sso_auth_keys WHERE expires_at < NOW()`

### 9. Add device fingerprinting

Collect and bind device details:

- Browser user-agent
- Screen resolution
- Timezone
- Installed fonts (optional, browser-side JS)
- Accept-Language header
- Combine all into a hash/fingerprint
- Store in auth_key payload

### 10. Add comprehensive logging

Log every auth attempt with:

- Timestamp
- Source product
- Target product
- User email
- IP address
- Device fingerprint
- Result (success or failure)
- Failure reason if any
- Auth_key hash (never log full token)

### 11. Add explicit error handling

Return clear messages for:

- Invalid auth_key format
- Auth_key expired
- Unauthorized product
- User not found
- Signature mismatch
- Device fingerprint mismatch
- IP mismatch (if enabled)
- HTTPS required

### 12. Add security options

Allow admins to configure:

- Token lifetime (default 30 seconds)
- Strict device binding (require exact match vs. allow tolerance)
- Strict IP binding (exact match vs. allow tolerance)
- Allow/disallow same-site re-authentication
- Rate limiting for failed attempts
- Enable/disable logging

### 13. Add testing

Create tests for:

- Valid auth_key login
- Expired auth_key rejection
- Invalid signature rejection
- Device fingerprint mismatch rejection
- IP mismatch rejection
- Product not trusted rejection
- User not found rejection
- Auth_key reuse (should work on subsequent request)
- Auto-cleanup after 30 seconds

## Security Rules

This plugin must always:

- use HMAC-SHA256 to sign all auth_keys
- use short-lived auth_keys, exactly 30 seconds expiry
- use device fingerprinting to bind auth_keys to the requesting device
- require HTTPS only
- validate source and target products
- treat auth_keys as potentially stolen (bind to device + IP + browser)
- store auth_keys in database only during the 30-second window
- auto-delete auth_keys after 30 seconds
- use constant-time comparison for HMAC verification
- log all failed authentication attempts
- never log full auth_keys or web_keys
- regenerate web_keys if any breach is suspected

## Implementation Notes

### Key Points

1. **30-Second Token Lifetime**: `auth_key` expires and must be auto-deleted from database after 30 seconds. Do not extend this window.

2. **Device Fingerprinting**: Collect browser, OS, screen resolution, timezone, and other device details. Hash them together. Reject if fingerprint changes on subsequent request.

3. **Re-use Optimization**: If the same `auth_key` is used again within 30 seconds on the same device, redirect immediately without re-validation (check database first).

4. **Auto-cleanup**: Use WordPress wp-cron or a direct cleanup query on every SSO request. After 30 seconds, auth_key must be deleted from the database.

5. **Web Key Security**: 
   - Each site generates its own `web_key` from an admin-provided PIN or auto-generated secret
   - Store `web_key` in WordPress options, never in database or config files
   - Allow admin to regenerate `web_key` if needed
   - Never log or expose `web_key` in error messages

6. **Constant-Time Comparison**: Always use `hash_equals()` for HMAC verification to prevent timing attacks.

7. **HTTPS Only**: Reject any non-HTTPS request at the `/sso` endpoint.

8. **Device Mismatch Handling**: If device fingerprint doesn't match, reject the auth_key and log it as a potential security incident.

9. **IP Address Tolerance**: Allow admin to configure whether IP must match exactly or if a tolerance is acceptable (e.g., same subnet).

10. **Logging**: Log auth attempts but never include full `auth_key` or `web_key`. Use hashes for sensitive data.

11. **Database Cleanup Strategy**: 
    - **Option A (Simple)**: Query and delete expired auth_keys on every `/sso` request
    - **Option B (wp-cron)**: Schedule a job to clean up every minute
    - Both are acceptable for 30-second TTL

12. **User Not Found**: If a user with the email doesn't exist, log it securely and show a generic error.

13. **Current User Logout**: Before logging in a new user, always log out any currently logged-in user on the target site.

14. **Redirect After Login**: After successful auth, redirect to the product's `page_url`, not to wp-admin or wp-login.

15. **Rate Limiting**: Optionally add rate limiting to the `/sso` endpoint to prevent brute-force attacks on auth_keys.

## Complete User Flow

### Setup Phase (Admin)

1. Admin on **Site A** goes to SSO settings
2. Admin generates a `web_key` using a PIN (stored in `product_based_sso_web_key` option)
3. Admin on **Site B** adds **Site A** as a product:
   - Product name: "Site A"
   - Logo: https://sitea.com/logo.png
   - Page URL: https://sitea.com/dashboard
   - Web Key: (paste the web_key from Site A's settings)
4. Same process: Admin on **Site A** adds **Site B** as a product with Site B's web_key

### User Switch Flow (Runtime)

#### **User on Site A clicks "Switch to Site B"**

1. **Site A (Source)**:
   - Verify user is logged in
   - Collect device fingerprint (user-agent, screen resolution, timezone, etc.)
   - Collect IP address
   - Build auth_key payload:
     ```json
     {
       "email": "user@example.com",
       "ip": "203.0.113.42",
       "device_fingerprint": "hash_of_device_details",
       "user_agent": "Mozilla/5.0...",
       "product_id": 2,
       "from_product": "site-a",
       "to_product": "site-b",
       "iat": 1714405200,
       "exp": 1714405230
     }
     ```
   - JSON-encode payload
   - Sign with HMAC-SHA256 using **Site B's web_key**
   - Create auth_key object: `{"payload": {...}, "signature": "xyz"}`
   - Base64-encode auth_key
   - **Redirect to**: `https://siteb.com/sso?auth_key=BASE64_ENCODED`

#### **User lands on Site B's /sso endpoint**

2. **Site B (Target)** - **First Request**:
   - Receive auth_key parameter
   - **Check database**: Is this auth_key already stored in `wp_sso_auth_keys`?
     - **If YES**: Jump to step 2.5 (immediate redirect, no re-validation)
     - **If NO**: Continue to step 2.1

   2.1 **Validate the auth_key**:
   - Confirm HTTPS
   - Base64-decode auth_key
   - Verify payload and signature exist
   - Recompute HMAC-SHA256 using **Site B's own web_key** (which matches Site A's web_key)
   - Use `hash_equals()` to compare signatures

   2.2 **Check token expiry**:
   - Confirm `exp >= time()` (not expired yet)

   2.3 **Check device binding**:
   - Collect current visitor's device fingerprint
   - Compare with device_fingerprint in auth_key
   - If mismatch: Reject and log as potential attack

   2.4 **Check IP binding** (if enabled):
   - Compare visitor's IP with auth_key's IP
   - If mismatch and strict mode: Reject
   - If mismatch and lenient: Allow if same subnet

   2.5 **Find user by email**:
   - Query WordPress users by email
   - If not found: Reject and log

   2.6 **Logout current user** (if any):
   - Call `wp_logout()`

   2.7 **Login the new user**:
   - Call `wp_set_current_user($user->ID)`
   - Call `wp_set_auth_cookie($user->ID, true)`
   - Trigger `do_action('wp_login', $user->user_login, $user)`

   2.8 **Store auth_key in database**:
   - Hash the auth_key (SHA256)
   - Insert into `wp_sso_auth_keys`:
     ```
     auth_key_hash: sha256(auth_key)
     user_id: $user->ID
     email: user@example.com
     ip_address: 203.0.113.42
     device_fingerprint: hash_of_device_details
     user_agent: Mozilla/5.0...
     source_product: site-a
     target_product: site-b
     expires_at: 2026-04-29 12:07:10 (now + 30 sec)
     created_at: now
     ```

   2.9 **Log the success**:
   - Insert into `wp_sso_logs` with event_type='success'

   2.10 **Redirect**:
   - Redirect to `https://siteb.com/dashboard` (the product's page_url)

3. **Site B (Target)** - **Immediate Re-request** (within 30 seconds):
   - User or system reloads/requests the same page with same auth_key
   - Query `wp_sso_auth_keys` for auth_key_hash
   - Found in database → **Immediately redirect to page_url without any validation**
   - This allows re-use within the 30-second window without expensive re-validation

#### **Auto-cleanup After 30 Seconds**

4. **Site B (Background)** - **Cleanup Job**:
   - Every minute (or on every request), run:
     ```sql
     DELETE FROM wp_sso_auth_keys WHERE expires_at < NOW()
     ```
   - Auth_keys older than 30 seconds are purged
   - Subsequent requests with that auth_key will be rejected

## Failure Scenarios

### Scenario 1: Invalid Signature
- User sends tampered auth_key
- HMAC verification fails
- Site B rejects and logs as 'invalid_signature'

### Scenario 2: Expired Token
- User takes more than 30 seconds to switch
- Site B checks `exp < time()` and rejects
- Logged as 'token_expired'

### Scenario 3: Device Mismatch
- User switches on browser A, then opens browser B with same auth_key
- Device fingerprint doesn't match
- Site B rejects and logs as 'device_mismatch'
- This is a security feature

### Scenario 4: User Not Found
- Auth_key is valid but email doesn't exist on Site B
- Site B rejects and logs as 'user_not_found'

### Scenario 5: Product Not Trusted
- Site B doesn't have Site A in its trusted products list
- Site B rejects and logs as 'untrusted_product'

## Security Considerations

- **Token Theft**: Even if auth_key is leaked, attacker cannot use it from a different device (device binding).
- **IP Spoofing**: Device fingerprint includes more than just IP (browser, OS, screen), making IP spoofing less useful.
- **Replay Attack**: Auth_key is deleted after 30 seconds, preventing indefinite reuse.
- **Time Sync**: Ensure all sites have synchronized clocks (NTP). If clocks are severely skewed, tokens may be rejected.
- **Web Key Rotation**: If web_key is compromised, admin can regenerate it. Old auth_keys will be rejected.
